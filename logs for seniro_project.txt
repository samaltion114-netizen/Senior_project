› EXPLAIN the ERD so I can understand the relationships you have made in the project


• ERD Overview
  The model is centered around one student’s learning journey:

  User -> Objective -> Task -> Session -> Proof -> (ProgrammingQuestion -> TodoItem)
  with side branches for AdminReview, Challenge, and AI audit/interview logs.

  Main Relationships

  1. User (custom auth model)

  - One User can be student/expert/trainer via flags.
  - One User can have:
      - one StudentProfile (1 : 0..1)
      - one ExpertProfile (1 : 0..1)

  2. StudentProfile and ExpertProfile

  - Both are OneToOne with User.
  - StudentProfile stores major, goal_text, weekly_availability, timezone.
  - ExpertProfile stores expertise_tags, bio.

  3. Objective

  - Objective.student -> User (many objectives per student).
  - A student can own many objectives (User 1 : N Objective).

  4. Task

  - Task.objective -> Objective (Objective 1 : N Task).
  - Each task belongs to exactly one objective.
  - Holds AI estimation output (estimated_minutes, estimation_confidence) and expected output fields for proof
    comparison.

  5. Session

  - Session.student -> User (User 1 : N Session)
  - Session.task -> Task (Task 1 : N Session)
  - A session is a scheduled time block for a task.
  - Status: planned/completed.

  6. Proof

  - Proof.session -> Session is OneToOne (Session 1 : 1 Proof max).
  - Proof stores uploaded image + explanation + ai_analysis.
  - This enforces one evidence record per completed session.

  7. ProgrammingQuestion

  - ProgrammingQuestion.proof -> Proof (Proof 1 : N ProgrammingQuestion)
  - Created when AI suspects mismatch/bug in proof.

  8. TodoItem

  - TodoItem.programming_question -> ProgrammingQuestion (1 : N)
  - Checklist items generated from suggested fixes.

  9. Challenge

  - Challenge.student -> User (User 1 : N Challenge)
  - Daily micro-tasks adapted to student issues/progress.

  10. AdminReview

  - AdminReview.proof -> Proof (Proof 1 : N AdminReview)
  - AdminReview.reviewer -> User (trainer user) (User 1 : N AdminReview)
  - Lets trainer confirm/override AI bug detection.

  - InterviewConversation.student -> User (User 1 : N InterviewConversation)
  - InterviewMessage.conversation -> InterviewConversation (1 : N)
  - Stores the AI interview flow and suggested objective payload.

  12. AIEventLog

  - AIEventLog.user -> User (User 1 : N AIEventLog, nullable)
  - Audit trail for prompts/responses/hash/embedding metadata.

  Why this structure works

  - It maps exactly to your workflow stages.
  - Keeps each stage normalized and auditable.
  - Separates operational data (objective/task/session/proof) from AI audit data (AIEventLog) and human governance
    (AdminReview).
  - Supports role-based behavior cleanly (student vs trainer vs expert).

  If you want, I can now draw a “plain-language ERD walkthrough” with 3 concrete sample records (student1 flow) so each
  FK becomes very obvious.
